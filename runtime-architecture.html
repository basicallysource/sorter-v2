The Four-Layer Runtime

Columns follow the physical machine (C1 bulk → C2 separation → C3 precise → C4 classification → Distributor). Only C2, C3 and C4 have cameras and perception pipelines; C1 and the Distributor run blind — C1 on timing + downstream capacity, the Distributor on piece handoffs. Stages inside each pipeline (Object Detector, Object Tracker, Filter Chain) are abstract protocols; concrete strategies are selected per feed via config. Runtimes are independent threads coordinating only via two typed protocols: PieceHandoff (solid red, downstream) and ReadySignal (dashed blue, upstream). Lifecycle, telemetry and side-effects ride the in-process EventBus; they never gate the hot path.

C4 ↔ Distributor — seven-step handshake

C4 owns the piece until it is ejected. The Distributor owns the bin decision. Neither side ever touches the other’s state directly — all coordination is typed messages on a bounded channel. A timeout at step 4 (bin unreachable) yields a HandoffAck(rejected) and routes the piece to the reject bin.

  1. Classify. C4 finishes classification and publishes PieceClassifiedEvent to the event bus. Telemetry only — the hot path does not wait.
  2. Request. C4 sends PieceHandoff.request(ticket, piece_summary) directly to Distributor. The Distributor FSM: idle → positioning.
  3. Move. Distributor consults the RulesEngine to resolve target_bin, then drives the chute servo.
  4. Grant. On servo-arrived, Distributor emits ReadySignal(capacity=1, ticket) → C4. C4 FSM: eject_pending → ejecting.
  5. Eject. C4 pulses the carousel stepper and waits the minimum settle time for the piece to leave.
  6. Commit. C4 sends PieceHandoff.commit(ticket, piece). Distributor FSM: positioning → sending → idle and publishes PieceSortedEvent.
  7. Ack. Distributor replies HandoffAck(ticket, accepted). C4 FSM: ejecting → idle; a slot becomes free and AdmissionStrategy picks it up on the next tick.

Strategy plugins — every swap point

Every amber plug in the diagram maps to one of these rows. All are looked up by string id in the StrategyRegistry, chosen per feed/runtime in recipe.toml. The goal is not to name a specific implementation; it is to name the contract that any implementation must honour.

Protocol What it does Examples (illustrative) Why swappable
ObjectDetector Turns a frame into a list of detections inside the feed’s zones. any learned or classical detector behind one detect(ctx) call Feed-specific detector tuning; new backbones swap in via config, never through if/else branches.
ObjectTracker Assigns stable IDs across frames and feeds motion prediction. motion-predictive trackers, cross-feed handoff trackers Tracker math stays on its own; only the interface changes. Anything with an update(ctx) fits.
TrackFilter Drops or annotates tracks after tracking (size, ghosts, regions, …). any chainable apply(ctx) stage, declared per feed in config Linear chain composed per feed. New filters drop in by registering an id.
Calibration Derives zones and homography for a feed at boot or on demand. manual polygons, fiducial boards, vision-driven auto calibration Calibration is a plug-in so manual tools and automated rigs co-exist.
Classifier Takes a piece dossier and returns identity (part + color). remote HTTP service, local ONNX model, multi-modal vision model Classifier is the one truly domain-specific piece. Swapping it enables coins, screws, any sortable.
RulesEngine Turns a classified piece into a target bin. tabular profiles, rule DSL, shape-only / color-only schemes Domain logic lives in the Distributor only. C4 never knows which bin a piece is going to.
AdmissionStrategy Decides when C4 can accept the next piece from C3. capacity counters, zone occupancy, arc-clear gates — composed Scattered if/else on admission becomes one swappable policy with one signature.
EjectionTimingStrategy Decides when C4 may release a piece onto the distributor. fixed delay, track-guarded, sensor-triggered Timing rules live in one object, testable in isolation from hardware.

Pinned decisions

Decided in the plan session. These are not up for debate during implementation — they narrow the search space so the rebuild can move.

Branch

Stay on sorthive

A pre-architecture marker commit on sorthive is the rollback reference. No second branch — avoids zombie paths. main stays untouched until the cutover PR.

Execution

Threaded actors

One thread per ComponentRuntime, one per per-feed pipeline, one per camera capture. Blocking stepper calls stall only their own runtime — never the main loop.

Coupling

Demand-pull via ReadySignal

Backpressure is the default, not a special case. Runtimes act only on a grant. Bounded queues (1–2) are transport; ReadySignal is the control.

Spur A · Perception

Carve & re-wire

The detector, tracker, handoff logic, burst store and calibration code have real substance — port them behind the new protocols, don’t rewrite them.

Spur B · Runtimes

Greenfield

The dual classification pipelines, blocking stepper calls and shared-state bag are not salvaged. coordinator.py, feeding.py, classification_channel/ are read as behavioral reference only.

Bin choice

Distributor owns it

C4 hands over a classified piece (part_id + color_id). The Distributor applies the RulesEngine to resolve the target bin. C4 never knows which bin is which.

Cameras

1:1 with active feeds

C2, C3 and C4 each get their own camera. C1 and the Distributor have no camera. SplitFeed remains in the codebase as a compatibility shim for legacy setups, but the target wiring is 1:1.

Cutover

Feature parity on merge

The PR that merges sorthive → main also deletes vision_manager.py, shared_state.py, coordinator.py, both old classification pipelines — in the same commit.

Config

Recipe + runtime vars

recipe.toml composes feeds, pipelines, runtimes and wiring. Live knobs (thresholds, pulse counts) stay in SQLite. Pydantic v2 validates both at startup.

Build sequence

Roughly 22–29 person-days. Phase 4 (RuntimeC4) is the riskiest — ~200 LoC of orchestration coordinating seven subsystems on real hardware with a concurrency-model change underneath.
