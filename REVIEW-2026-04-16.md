## 2. Executive Summary

**Product shape:** Two SvelteKit frontends (sorter local UI + Hive community), two Python/FastAPI backends, Raspberry Pi Pico firmware (2 role variants), Jekyll docs. Solo-ish maintainer scope.

**Overall health:** Working system with real hardware integration and a thoughtful SortHive split. **Not production-hardened.** Main risks:

1. **Local backend security** ‚Äî sorter API fully open (CORS `*`, no auth). Defensible IF strictly LAN-only monitoring; serious the moment it's port-forwarded or a Pi is stolen.
2. **Frontend maintainability wall** ‚Äî megacomponents, two diverging UI component libraries. Debt compounding.
3. **Ops immaturity** ‚Äî no CI, fragile install.sh failure modes, unhardened systemd units, no observability, docs drift.

**Strategic question:** Personal lab + showcase, or shipped-to-other-makers? Current state is halfway. Answer determines how much of below is worth doing.

## 3. Top 15 Findings ‚Äî Tracker

### P0 ‚Äî Fix before any external user runs this

- [x] **1. Sorter API has no auth + CORS `*`** ‚Äî bind lockdown done in `46085c5`. CORS narrowed to derived origins. **Still open:** bearer-token auth (anyone who *is* allowed to hit the API can still hit every endpoint unauthenticated). Revisit if exposing on LAN regularly.

  - *Mitigation:* Fernet-encrypt with key derived from machine-id; document threat model.

## 4. Strategic Observations (PM-level)

- **Scope vs throughput.** Mechanical + electronics + firmware + vision ML + two web apps + docs. One maintainer cannot harden all of it. Pick:
  - **(A) Personal lab + open-source showcase.** Accept rough edges. Focus on hardware + ML quality. Mark ops/security "LAN-only, single user".
  - **(B) Ship to other makers.** Commit to CI, install reliability, security baseline, docs accuracy. Cut features if needed.
  - Codebase today is halfway. Pick and name it.

- **Sorter ‚Üî Hive split is correctly drawn** but the **contract layer is missing**. Both sides invent their own payload schemas (`server/sample_payloads.py` on sorter, `app/services/sample_payloads.py` on hive). One shared JSON Schema ‚Üí codegen both sides. Prevents the silent-breakage class of bug this split is most vulnerable to.

- **Two frontends without a shared package is a regret in waiting.** Every week they diverge more. Extract `software/ui/` before Hive grows.

- **"Simplify profile lifecycle" memory vs 1342-line profiles page** ‚Äî direct contradiction between stated direction and current reality. (Partially addressed by split in `c0efb29`, but lifecycle simplification still pending.)

- **Feature gaps (what users expect, missing today):**
  - Profile version diff UI ("what changed between v1 and v2")
  - Bulk sample review (approve/reject N at once)
  - Machine config export/import (for hardware swap)
  - Real-time metrics dashboard (sort rate, accuracy trend)
  - Undo on profile activation

- **Hardware-aware UX gaps:**
  - Hardware error banner has multiple entry points with unclear difference ("Refresh Status" vs "Retry Home")
  - ArUco calibration has no user-facing route ‚Äî if cameras move, users stuck
  - Camera setup bifurcated between setup wizard and settings page
